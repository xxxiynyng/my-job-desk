너는 지금부터 Pickd(My Job Desk) 레포에서 **자율주행 모드**로 MVP 정비 작업을 수행한다. 사용자는 자리를 비우며, 권한 요청은 승인될 것이다. 아래 규칙과 5단계를 순서대로, 게이트를 지켜 진행하라.

## 0. 먼저 읽고 시작 (컨텍스트 고정)

1. 루트 `CLAUDE.md`를 정독한다. 이 파일이 코드/작업 규칙의 단일 출처다. 특히 **디자인 일관성 원칙**(임의로 새 색·굵기·간격 값 만들지 말 것, 기존 토큰·공용 컴포넌트 재사용), **진단 규칙**(추측 금지, 코드에서 원인 확인, 정렬·간격은 수치로 검증), **커밋 규칙**(항목별 개별 커밋 = 되돌림 단위, `type(scope): 한국어 설명`)을 반드시 지킨다.
2. 디자인/기획 SSOT는 Notion이다. 핵심 가치는 **"경험의 데이터 자산화"** — 경험을 한 번 구조화하면 모든 지원서·자소서에 재사용된다는 것. 모든 UI/서비스 판단은 이 가치에 부합해야 한다.
3. **⚠️ 중요 — 오래된 보고서를 그대로 믿지 말 것.** 레포에 `MVP_AUTORUN_LOG.md`, `SPRINT_REPORT.md`, `UPDATE_REPORT.md`가 있다. 이건 이전(2026-06-29~07-01) 자율주행 기록이다. 그 뒤 2026-07-02 결정으로 코드가 더 진행됐다. 예를 들어 그 보고서들은 "상태 레이블 7개 vs 기획 9개 불일치(Critical)"를 미해결로 적고 있지만, **현재 코드는 이미 6단계 체계(작성중/지원완료/서류전형/필기전형/면접전형/전형완료 + finalResult 배지)로 재편 완료**되어 그 이슈는 해소된 상태다. 따라서 이 보고서들은 "과거 참고용"으로만 읽고, **진단은 반드시 현재 코드 + 최신 SSOT를 직접 대조**해서 다시 하라. 이미 해결된 항목을 다시 문제로 올리지 말 것.

## 1. 작업 원칙 (자율주행 가드레일)

- **직접 코드를 읽고 원인을 확인한 뒤 수정한다.** grep으로 정확한 위치를 짚고, 추측으로 고치지 않는다.
- **기존 디자인 시스템을 유지·활용한다.** 어색하면 ① SSOT 토큰 확인 → ② 기존 잘 된 컴포넌트(예: 탭1 공고명, `components/table/*`, `components/pickd/ds/*`)를 재사용 → ③ 새 스타일 발명 금지. "통일"은 잘 된 쪽에 맞추는 것이지 양쪽을 새 값으로 바꾸는 게 아니다.
- **항목별 개별 커밋**으로 남긴다(되돌림 단위). 한 커밋에 여러 무관한 변경을 섞지 않는다.
- **모든 변경 후 자가검증**: `npx tsc -p tsconfig.app.json --noEmit`(신규 에러 0) + `npm run build` 통과 + `npm run test` 통과. 실패하면 다음 항목으로 넘어가지 말고 고친다.
- **자율주행 범위를 벗어나는 것**(아래)은 **직접 바꾸지 말고 최종 보고에 "사용자 결정 필요"로 남긴다**:
  - localStorage 저장 구조를 깨는 변경(기존 사용자 데이터 손실 위험). 꼭 필요하면 키 버전 suffix를 올리고(`.v2` 등) 마이그레이션 방안을 제안만 한다.
  - 기획 자체가 미확정인 항목(예: 중복 경험 병합 유사도 임계값, 자격증 가산점 자동계산 채택 여부).
  - 백엔드 API 연결이 필요한 부분(현재는 프론트 전용 프로토타입, mock 데이터). mock을 실제 fetch로 바꾸지 않는다.
- 데이터가 없어 판단이 안 서면, 임의로 만들지 말고 보고에 질문으로 남긴다.

## 2. 현재 단계 (내가 파악한 진단 — 이걸 출발점으로 검증하라)

- **탭1 지원 대시보드**: MVP 완료. 6단계 전형 체계 반영됨. 프론트 전용(mock 데이터, 실 API 미연결).
- **탭2 경험·스펙 DB**: 집중 개발 중. `src/pages/Experiences.tsx`가 가장 큰 파일. 13개 프리셋·기본정보·파일함·대표스펙 구조.
- **탭3 AI 자기소개서**: 기획/백로그. `src/pages/AICover.tsx`는 뼈대 수준.
- 프로토타입 배포: pickd-seven.vercel.app (main push 시 자동 배포).
- 이전 자율주행이 UI/UX·서비스·클린업 감사를 1차로 수행함(위 3개 보고서). 이번엔 **그 이후 상태에서 다시** 점검한다.

## 3. 수행 절차 — 5단계 (게이트 준수: 앞 단계 통과 후 다음)

### Phase 1 — 프로토타입 동작 확인 (게이트: 여기 통과해야 Phase 2 진입)
1. `npm install` → `npx tsc -p tsconfig.app.json --noEmit` → `npm run build` → `npm run test`를 순서대로 실행한다.
2. 하나라도 실패하면 원인을 코드에서 찾아 고치고(개별 커밋), 다시 전부 통과시킨다.
3. 라우트 구조(`/`, `/jobs/:slug`, `/experiences`, `/ai-cover`, `/settings`, `/calendar`)가 정상 로드되는지 소스 기준으로 확인한다.
4. **결과를 한 줄로 확정**: "Phase 1 전부 통과" 여부. 통과 못 하면 Phase 2로 넘어가지 않는다.

### Phase 2 — UI/UX 감사 & 수정 (최고의 UI/UX 방법론 적용)
목표: 오류 및 **"경험의 데이터 자산화" 가치에 반하는 UX**를 찾아 고친다. 점검 렌즈:
- **일관성**: 색·간격·타이포·상태 배지·칩·테이블 규칙이 SSOT 토큰과 일치하는가. 같은 요소가 화면마다 다른 스타일로 렌더되지 않는가(정렬·간격은 눈대중 금지, 좌표/수치로 검증).
- **접근성**: icon-only 버튼·링크 aria-label, 키보드 포커스(focus-visible) 표시, 삭제 등 파괴적 액션의 확인 다이얼로그와 경고 문구.
- **카피 톤**: 토스트/문구는 `~어요` 체 통일(예: "저장됐어요"). AI 관련 문구는 "부족/미흡/불완전" 같은 부정 표현 금지 → "더 구체적으로 만들어 봐요" 류 권장.
- **가치 정합성**: 경험 정리가 일회성이 아니라 누적 자산으로 느껴지는 흐름인가. 원본 경험(탭2)과 지원용 가공(탭1 지원준비)의 분리가 UI에서 흐트러지지 않는가.
각 발견 항목은 원인→수정→개별 커밋으로 처리하고, 판단이 필요한 큰 변경은 보고로 남긴다. Phase 종료 시 tsc/build/test 재검증.

### Phase 3 — 서비스 개발 감사 & 수정 (최고의 서비스 개발 방법론 적용)
목표: 오류 및 **서비스 로직이 아이템 가치에 반하는 지점**을 찾아 고친다. 점검 렌즈:
- **불변 원칙 준수**: AI는 사용자 문장을 자동 수정·삭제하지 않는다 / AI 생성문은 "반영" 또는 드래그로만 삽입된다 / 파일 추출 결과는 체크리스트 승인 전까지 미확정 / 자동 병합 금지. 코드가 이를 어기는 곳이 없는지 확인.
- **상태·데이터 일관성**: 전형 6단계 + finalResult(합격/불합격/보류) 체계가 관련 파일 전체에서 일치하는가(`StatusBadge`, `JobPostingTable`, `RowContextMenu`, `StatusManagementModal`, calendar). 서로 다른 상태 집합이 두 세트로 공존하지 않는가.
- **안전/정합성**: 외부 링크 `noopener,noreferrer`, D-DAY 계산(자정 기준), localStorage 키 버전 관리, 타입 안전성(any 남용 제거).
- **경계 준수**: 백엔드 없는 프로토타입 전제. 없는 API를 호출하거나 서버 로직을 넣지 않는다.
개별 커밋 + Phase 종료 시 재검증.

### Phase 4 — 불필요 코드 정리
- 미사용 import·변수·함수·컴포넌트(dead code), 렌더되지 않는 컴포넌트, 남은 `console.log`, 주석 처리된 죽은 코드 제거.
- 중복 로직은 기존 공용 컴포넌트로 수렴(예: 테이블 관련은 `components/table/*` 재사용). **새 추상화를 발명하지 말고** 이미 있는 공용으로 묶는다.
- 삭제가 동작에 영향 없는지 tsc/build/test로 확인 후 개별 커밋.

### Phase 5 — 정리·보고
1. 전체 최종 검증: tsc(신규 에러 0) + build + test 통과 확인.
2. 변경 요약 보고서를 `AUTORUN_REPORT_2026-07-05.md`로 새로 작성한다(기존 보고서 덮어쓰지 말 것). 포함 내용:
   - Phase별 표: 항목 / 조치(수정·이미정상) / 커밋 해시 / 비고.
   - **⚠️ 사용자 결정 필요** 섹션: 자율주행 범위를 벗어나 손대지 않은 것(데이터 구조 변경, 기획 미확정, API 연결 필요 항목)과 권장안.
   - **되돌리는 법**: 개별 커밋 `git revert <hash>`, 세션 전체는 시작 태그로 `git reset --hard`.
3. push는 하지 않는다(사용자가 확인 후 직접). 다만 로컬 커밋은 모두 남겨둔다.
